# 三层可行性分层表

本文件是当前"课程蓝图收口"的核心文档。

目标：
- 把课程层、玩家层、当前实现层彻底切开
- 每个单元明确标记：保留 / 降级 / 近期可做 / 远期冻结
- 避免课程蓝图继续跑到远超引擎能力的地方而不自知
- 为下一阶段的实装提供可执行判断依据

---

## 0. 三层定义

### 课程层（Curriculum）
教学设计视角。允许超前，允许假设新交互。
描述的是"如果资源无限，课程应该怎么排"。

### 玩家层（Player）
孩子和家长实际看到什么。
只包含"当前已实装或近期一定能实装"的内容。
对应 UI 的导航、章节、阶段名。

### 实现层（Implementation）
当前引擎真正能承载什么。
只有两种关卡类型：`solve` 和 `build`。
不具备：观察关、判断关、半建模关、guided-workshop、mixed、线段图视图。

## 3. 关键决策记录

### 3.1 B1 / B2 合并
- **原因**：chapter_blueprints_master.md 中 B1 和 B2 使用完全相同的 3-1、3-2、3-3 关卡，只是标签不同。这是重复计数，不是真正的两个教学单元。
- **做法**：合并为一个单元"B1 未知数的倍数与平均分"，保留 3 关。
- **影响**：第二程少一个子章节，但不影响关卡数量。

### 3.2 D1 已正式落位
- **原因**：叙事压缩稿把 D1 放进第二程后，第二程已包含 A-Bridge + B1(含B2) + B3 + B-Bridge + D1，节奏过长。
- **做法**：已将 D1 从第二程移出，并正式落位为“第三程与第四程之间的独立过渡段”。
- **不做**：不再把 D1 视作待定项，也不把它强行塞进第四程主体。

### 3.3 D3 并入 D4
- **原因**：D3 只有 2 关，且原蓝图要求 mixed 类型交互，引擎不支持。教学职能（方法选择回顾）有价值，但不值得为此造新系统。
- **做法**：课程层保留"方法地图复习"的教学节点；产品层第一版把它折叠进 D4 线段图桥接的尾声阶段结算页。
- **注意**：当 D4 本身也是远期冻结时，D3 同样冻结。近期实现中不需要单独处理。

### 3.4 A0 的近期实现策略
- **原因**：A0 的教学职责（等号 = 两边一样多）是主线最重要的起点，但原蓝图要求的"观察关/判断关/轻按钮关"都不是现有棋盘系统。
- **做法**：近期用 solve 变体 + 教程引导近似。设计 2-3 个极简关卡，等式已经平衡或只需一步操作，文案和教练提示全部围绕"一样多"而不是"移项"。
- **红线**：这只是临时替身，不是课程层替代。课程层 A0 继续保留原蓝图的教学目标。

### 3.5 C3 的近期实现策略
- **原因**：C3"选择未知量"是 build 关之前的关键桥梁，但原蓝图要求半建模交互。
- **做法**：近期用 build 关变体近似——在 build 关的文案/故事卡中先显式问"谁是未知量"，用教练提示引导选择，然后再进入正常 build 流程。
- **红线**：不可膨胀成新系统。近期实现就是"build 前加一小步选择"。

### 3.6 C4 和 D4 远期冻结
- **原因**：C4 要求 guided-workshop 流程，D4 要求线段图视图，都不是当前引擎能力。
- **做法**：课程层文档保留，实现层和玩家层明确标记"远期冻结"，不进入近期实装讨论。

### 3.7 ch13-15 远期冻结
- **原因**：挑战线当前 4 章 20 关已经够用，扩展假设了更多新交互，不该混进近期路线。
- **做法**：冻结。
